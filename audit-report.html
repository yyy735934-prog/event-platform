1. 项目概览与技术栈

后端 (Worker)
  • 运行时: Cloudflare Workers
  • 框架: Hono
  • 数据库: Cloudflare D1 (SQLite)
  • 会话: Cloudflare KV (7 天 TTL)
  • 密码: PBKDF2-SHA256, 100k 迭代
  • 邮件: Resend API
前端
  • 框架: Vue 3 + Vue Router (History 模式)
  • 构建: Vite (双入口: web/ + admin/)
  • 状态管理: 无 (localStorage + 组件内 ref)
  • 样式: 纯 CSS, CSS 变量主题
  • QR 码: qrcode npm 包
  • 扫码: BarcodeDetector API (浏览器原生)

2. 活动生命周期

草稿 draft → 待审核 pending → 报名中 open → 进行中 active → 已结束 closed
  • 提交审核: draft → pending (主理人操作)
  • 审核员自己的活动: draft → open (跳过 pending, 自动批准)
  • 审批通过: pending → open (审核员操作, 触发邮件)
  • 驳回: pending → draft (审核员操作, 附驳回原因, 触发邮件)
  • 撤回: pending → draft (主理人自行撤回)
  • 开始活动: open → active
  • 结束活动: active → closed
  • 复制活动: 任意状态 → 生成新 draft (标题加 "(副本)")
  • 删除活动: 仅限 draft 且无报名记录

10. 发现的问题与风险

安全 / 隐私问题
  1. 参与者邮箱查询无限制: GET /api/participant/my-events?email=xxx 无需认证, 且返回包含 token 字段, 意味着知道任意邮箱就能获取该人的所有 QR 签到码。
  2. 公开签到接口无认证: POST /api/signups/checkin-by-token 和 POST /api/signups/checkin 完全公开, 理论上可被批量调用进行虚假签到。
功能缺陷
  1. 手动添加报名不检查容量: POST /api/signups/manual 允许管理员添加报名时不受容量限制, 可能导致超额。
  2. 签到确认邮件是死代码: checkinConfirmEmail 已定义并导入但从未调用, 签到不会发送任何邮件通知。
  3. schema.sql 严重滞后: 7 个列在生产代码中使用但 schema.sql 未更新, 新部署需手动添加。
架构观察
  1. 无共享组件: web/ 和 admin/ 均无 components/ 目录, 所有 UI 在 views 内自包含, 代码复用有限。
  2. 无状态管理库: 仅使用 localStorage + 组件内 ref, 对当前规模足够。
  3. 事件日期为自由文本: event_date 是 TEXT 类型, 无日期格式校验。
  4. 邮件发送静默失败: email.js 捕获所有异常且不抛出, 邮件发送失败不会报错。
